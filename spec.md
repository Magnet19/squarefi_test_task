# Тестовое задание: Next.js E-Commerce

## 📋 Описание

Разработать веб-приложение на **Next.js (App Router)** с авторизацией и автоматическим обновлением JWT токенов.

**Backend API:** [DummyJSON](https://dummyjson.com/docs/auth)

---

## 🎯 Функциональные требования

### Страницы

| Страница | Путь | Доступ |
| --- | --- | --- |
| Root | `/` | Редирект |
| Login | `/login` | Публичная |
| Dashboard | `/dashboard` | Защищённая |

### Root (`/`)

- Если у пользователя есть токены → редирект на `/dashboard`
- Если токенов нет → редирект на `/login`

### Login (`/login`)

- Форма авторизации (username/password)
- Тестовые данные указаны в документации к API
- При успехе — редирект на Dashboard
- При ошибке — отображение сообщения об ошибке

### Dashboard (`/dashboard`)

### При открытии страницы должны быть доступны следующие данные (загружаем по SSR):

| Данные | API |
| --- | --- |
| Информация о пользователе | `GET /auth/me` |
| Корзины пользователя | `GET /auth/carts/user/{userId}` |
| Первые 5 продуктов | `GET /auth/products?limit=5` |

### По действию пользователя:

**Пагинация продуктов:**

- Кнопка "Загрузить ещё" — подгружает следующие 5 продуктов
- Новые продукты добавляются к существующему списку

**Добавление товара в корзину:**

- У каждого продукта кнопка "В корзину"
- При клике — отправка запроса `POST /auth/carts/add`
- Отображение результата (успех/ошибка)

**Кнопка "Выйти":**

- Очищает токены
- Редиректит на `/login`

## 🔄 Требования к Token Refresh

### Основное поведение

- Если access token недействителен (истёк или невалиден) — автоматически обновить токены
- Если для корректной работы приложения нужно выполнить упавший запрос к API повторно — это должно происходить без участия и информирования пользователя
- При невозможности обновить токены — логаут и редирект на `/login`

### Single Refresh

При параллельных запросах, если для всех требуется обновление токена — запрос на refresh должен выполняться **только один раз**, а не для каждого запроса.

### Изоляция между сессиями

Логика refresh должна быть изолирована между разными сессиями — refresh одного пользователя не должен влиять на других пользователей или другие сессии того же пользователя.

## 📐 Требования к архитектуре

- Взаимодействие с API через серверную часть Next.js (API Routes / Server Actions / Server Components). Прямое взаимодействие браузер → API не допускается.
- При проектировании API модулей необходимо учесть, что количество эндпоинтов в будущем будет значительно расширяться. Решение должно легко масштабироваться без необходимости переписывать базовую логику.

## 📚 API

**Base URL:** `https://dummyjson.com`

**Важно:** Для работы с защищёнными данными необходимо использовать эндпоинты с префиксом `/auth/`. Такие эндпоинты требуют Bearer token в заголовке Authorization и возвращают 401 при невалидном/просроченном токене.

| Эндпоинт | Метод | Описание |
| --- | --- | --- |
| `/auth/login` | POST | Авторизация, получение access и refresh токенов |
| `/auth/refresh` | POST | Обновление токенов |
| `/auth/me` | GET | Данные текущего пользователя |
| `/auth/products` | GET | Список продуктов (поддерживает `limit` и `skip`) |
| `/auth/carts/user/{id}` | GET | Корзины пользователя |
| `/auth/carts/add` | POST | Добавление товара в корзину |

**Документация:**

- [Auth](https://dummyjson.com/docs/auth) — авторизация и токены
- [Products](https://dummyjson.com/docs/products), [Carts](https://dummyjson.com/docs/carts) — структура данных, параметры запросов и примеры ответов (эндпоинты без `/auth/` префикса публичные, но описание применимо и к защищённым)
